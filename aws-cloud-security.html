AWS Cloud Security Case Study

Designing and implementing a secure AWS environment with a focus on identity, network segmentation, logging and risk-based control selection.

1. Overview

This lab-style project was designed to mirror how a small organisation might establish a secure AWS footprint. The goal was to apply cloud security principles in a structured way, linking each design choice back to risk, governance and operational requirements.

2. Scope & Assumptions

  • Single AWS account representing a small organisation or business unit.
  • Mix of internet-facing and internal workloads.
  • Focus on core security controls rather than advanced cloud-native services.
  • Emphasis on auditability, traceability and least-privilege access.

3. Architecture & Design

I designed a basic but secure reference architecture that can be extended over time. Key design elements included:

  • VPC with separate public and private subnets.
  • Internet-facing resources placed only in public subnets where required.
  • Application and data components hosted in private subnets.
  • Security groups limiting inbound and outbound traffic based on role.
  • Use of IAM roles for services instead of long-lived access keys.

The architecture was documented so that each component could be explained back to a non-technical stakeholder in terms of risk reduction and resilience.

4. Identity & Access Management

IAM was treated as a core control rather than an afterthought. I:

  • Defined roles for administrators, developers and read-only users.
  • Applied least-privilege policies mapped to specific job functions.
  • Used IAM roles for EC2 and services instead of embedding credentials.
  • Documented which roles had access to which resources and why.

This allowed me to show how access design links directly to governance and audit expectations.

5. Logging, Monitoring & Evidence

To support investigation and assurance activities, I:

  • Enabled CloudTrail for account-level activity logging.
  • Ensured logs were centralised and retained for review.
  • Verified that key events such as console logins and IAM changes were captured.
  • Captured screenshots and configuration details as project evidence.

The focus was on being able to demonstrate “who did what, where and when” in a way that aligns with audit and incident response requirements.

6. Risk-Based Decisions & Next Steps

Throughout the build, I noted decisions, trade-offs and residual risks. For example:

  • Which services were in scope for this phase and which were deferred.
  • Where default configurations were acceptable and where hardening was needed.
  • What controls would be prioritised in a follow-on phase (e.g. WAF, GuardDuty).

Future enhancements could include integration with a SIEM, automated compliance checks and more detailed threat-modelling for critical workloads.
