SIEM Visibility & Detection Case Study

Establishing security visibility through log collection, detections and a simple incident response workflow, with a focus on linking alerts back to business risk and operational impact.

← Back to Projects

1. Overview

This project focuses on building practical experience with a Security Information and Event Management (SIEM) platform. The goal is to move from raw log data to meaningful alerts and investigations, using a structured, risk-aware approach that mirrors how SOC and security teams operate in practice.

2. Scope & Assumptions

  • Small set of systems (e.g. one Windows host, one Linux host or lab VMs).
  • Focus on authentication, system activity and basic security events.
  • Objective is understanding flow: log → event → alert → investigation.
  • Assumes a hosted or lab SIEM platform is available for testing.

3. Log Sources & Onboarding

I began by selecting which log sources would add the most value for a small environment, then onboarded them into the SIEM:

  • Authentication logs (logon success/failure, account lockouts).
  • System logs for key services and security-relevant events.
  • Optional: basic firewall or endpoint security logs if available.

The aim was to ensure that any investigation into a suspicious login or system change would have enough supporting detail to be meaningful.

4. Detections & Use Cases

I then defined a set of simple but realistic detection use cases, such as:

  • Multiple failed logins followed by a successful login from the same source.
  • Logins outside expected working hours.
  • New local user creation on a server or critical host.
  • Service or process execution that is unusual for that system role.

For each use case, I created detection rules or queries in the SIEM and validated that they would trigger under the right conditions.

5. Dashboards, Triage & Investigation

To support ongoing monitoring and incident handling, I:

  • Built simple dashboards to show failed logins, account lockouts and key security events.
  • Configured alert views for quick triage.
  • Performed test investigations using simulated suspicious activity.
  • Documented the steps taken from initial alert through to basic root cause analysis.

This helped connect the technical view (logs and alerts) with the operational view (what this means for the organisation).

6. Incident Workflow & Lessons Learned

I outlined a lightweight incident handling workflow covering:

  • Alert review and initial triage.
  • Enrichment and gathering of additional context.
  • Escalation criteria and potential containment steps.
  • Documentation of findings and recommendations.

I also captured lessons learned around noise reduction, alert tuning and the importance of understanding normal behaviour before defining what is “suspicious”.

7. Next Steps

Future enhancements to this lab could include:

  • Integrating additional log sources such as cloud platforms or SaaS tools.
  • Building more advanced correlation rules across multiple systems.
  • Linking detections to a formal playbook or SOAR-style automation.
  • Mapping detections to frameworks such as ATT&CK for coverage analysis.

Overall, this project demonstrates how I approach visibility, detection and incident handling using a structured and risk-based perspective, rather than focusing purely on tooling.
