Evidence Pack (ACS Secure AWS)

This page consolidates screenshots showing what was configured, why it matters (risk/threat/vulnerability), and what action was taken. Each item is written as if you’re explaining it to a hiring manager in a real-world environment.

Network security & segmentation

VPC design, public/private subnets, controlled ingress, and visibility through flow logs.

Risk Network exposure Action taken

Public vs private subnet separation

Designed subnets to limit blast radius and keep application components private where possible.

ThreatDirect internet access to internal workloads
DecisionUse private subnets for non-public tiers; public subnet only for what must be reachable
Control Public IP behaviour Action taken

Public subnet configured for auto-assign public IPv4

Public workloads require an internet-routable address; this was enabled only for the public subnet.

VulnerabilityMisconfigured subnets can accidentally expose private instances
DecisionEnable public IPv4 assignment only in the public subnet
Control Internet egress/ingress path Action taken

Internet Gateway attached to the VPC

Provides the controlled internet path for public subnet routing (without implying everything is public).

ThreatUncontrolled outbound/inbound traffic paths
DecisionUse IGW + routing only for public subnet; keep private tiers isolated
Visibility Detection & forensics Action taken

VPC Flow Logs enabled (to CloudWatch Logs)

Captures accepted/rejected network flows to support investigations and troubleshooting.

RiskLimited audit trail of network activity during incidents
DecisionEnable Flow Logs to CloudWatch for searchable, timestamped traffic metadata
Risk Brute force / SSH exposure Action taken

Restricted SSH access (admin only)

SSH is limited to a specific trusted source rather than open internet access.

ThreatBrute-force attempts and credential stuffing against SSH
DecisionAllow TCP/22 only from a known IP (admin access), not 0.0.0.0/0
Control Access path Action taken

Bastion access workflow validated

Demonstrates operational access method and that remote admin access is intentionally controlled.

RiskAd-hoc access routes to private resources
DecisionUse a controlled access path (bastion pattern) rather than opening private tiers

IAM hardening

Strong authentication, reduced privilege, and account access controls.

Risk Account takeover Action taken

MFA enabled

MFA reduces the likelihood that stolen credentials alone lead to full account compromise.

ThreatPhishing / leaked credentials
DecisionEnforce MFA for privileged access to reduce takeover risk
Control Privileged access Action taken

Admin group protected with MFA

Privileged groups are protected with stronger authentication requirements.

RiskPrivilege escalation if admin identities are compromised
DecisionEnsure admin access is tied to MFA and controlled membership
Control Password policy Action taken

Strong password policy enforced

Improves resistance to brute force and weak-password attacks.

VulnerabilityWeak or reused passwords
DecisionEnforce complexity/length requirements aligned to good practice

Logging & monitoring

Audit trails for governance + operational signals for detection and investigation.

Detection Audit trail Action taken

CloudTrail enabled

Records API actions for accountability, incident response, and compliance evidence.

RiskNo reliable record of “who did what” during a security incident
DecisionEnable CloudTrail so changes to IAM, networking, and services can be traced
Monitoring Log group events Action taken

CloudWatch Logs group receiving events

Centralises logs for visibility, alerting, and investigation workflows.

ThreatAttacks can go unnoticed without central monitoring
DecisionSend logs to CloudWatch to make events searchable and alertable

Compliance & governance

Policy-as-code style checks and continuous monitoring for drift and misconfiguration.

Governance Config rules Action taken

AWS Config rules in place

Provides automated checks against common misconfigurations and security drift.

RiskSecurity controls weaken over time due to change/drift
DecisionUse AWS Config rules to flag non-compliance early and consistently

Cost management & billing visibility

Controls to prevent surprise spend and maintain transparency.

Risk Uncontrolled spend Action taken

Cost & usage reporting (CUR)

Provides a detailed breakdown of spend by service for analysis and optimisation.

ThreatBudget creep and unexpected costs
DecisionEnable CUR for accurate reporting and cost governance
Visibility Dashboard Action taken

Security & Billing dashboard

Single pane view that combines security/compliance signals with billing awareness.

RiskSecurity and cost data viewed in silos
DecisionCreate a dashboard to monitor both security posture and spend indicators

Evidence the web tier actually works after being deployed and secured.

Apache installed and enabled

Shows the web server was installed and configured to start and run as a service.

Risk“Paper architecture” with no functional workload proof
